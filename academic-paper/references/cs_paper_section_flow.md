## 基本方針

論文全体では、Introduction で示した研究目的と主張を軸にする。各章は、その目的と主張を支えるために配置する。章ごとの内容を機械的に並べてはならない。背景、目的、問題、提案、実験、結果、考察が自然につながるようにする。

執筆時・レビュー時に常に確認する：

- 研究の社会的・技術的背景が、読者と共有できる形で示されているか。
- 研究の目的が早い段階で明確になっているか。
- 対象とするタスクや問題が具体的に説明されているか。
- その問題がなぜ難しいのか、既存手法がどのような場合に不十分なのかが説明されているか。
- 提案手法が、その問題に対する自然な解決策として導入されているか。
- 提案手法と既存手法の違いが、主要な差分に絞って説明されているか。
- その差分を導入する理由が説明されているか。
- 実験が、Introduction や Method で述べた主張を検証しているか。
- 結果から何がわかったのかが明確に書かれているか。
- 失敗例や限界が、単なる欠点ではなく、今後の改善につながる知見として説明されているか。
- 各章が孤立せず、論文全体の流れに貢献しているか。

## 論文の基本構成

ロボット応用を中心とする CS 論文では、原則として以下の構成を想定する。

```
1. Introduction
2. Related Work
3. Problem Setting / Problem Statement
4. Proposed Method / Method
5. Experiments / Experimental Setup
6. Results / Experimental Results
7. Discussion
8. Conclusion
```

**重要:** 章構成はあくまで出発点であり、守ることが目的ではない。読者が研究の目的・位置づけ・解決方針・検証結果・限界を自然に理解できる流れを作ることが本質である。そのために必要な章を置き、不要な章は省いてよい。

### 章を省略・統合してよい条件

研究が扱うタスクや問題が読者に広く知られている場合、独立した Problem Setting の章を設ける必要はない。タスクの定義や前提は、実験の冒頭（Experimental Setup）で簡潔に触れれば十分なことが多い。

**例：物体認識タスク**
物体認識のような標準的なタスクを対象とし、既存のベンチマークデータセットで精度評価する場合、「このデータセットで何クラスの分類を行い、Top-1 accuracy で評価する」という内容は Problem Setting として独立した章にする必要はない。Experimental Setup の冒頭に書けばよい。Problem Setting の章が必要になるのは、タスク自体の定義が自明でない場合、入力・出力・成功条件を形式的に定義しないと Method が理解できない場合、などである。

同様に、Research Content や投稿先の慣習によって章名や分け方も調整してよい。たとえば Results と Discussion を統合した "Results and Discussion" とする場合や、Experimental Setup と Results をまとめた "Experiments" とする場合もある。

## 各章の役割と流れ

### 1. Introduction

Introduction では、研究領域の背景を述べた後、この論文が何を実現したいのかを早い段階で明確にする。目的を述べる前に、既存研究の詳細や個別手法の説明に入りすぎない。

Introduction の冒頭は、読者と共有できる最大公約数的な事実から始めるとよい。多くの読者が自然に同意できる背景を示したうえで、その中にある未解決の問題へと話を進める。

読者が以下の流れで理解できるように構成する：

1. なぜこの研究領域が重要か。
2. 本研究では何を実現したいのか。
3. 対象とするタスクや問題は何か。
4. そのタスクや問題の具体例、ユースケースは何か。
5. その問題はなぜ難しいのか。
6. 既存手法群はなぜ不十分なのか。
7. 本研究では何を提案し、何を解決するのか。
8. 提案手法は既存手法と何が違うのか。
9. その違いを導入する理由は何か。
10. どのような評価によって、何が明らかになったのか。
11. 本研究の新規性や貢献は何か。

この順序は、本文中で明示的なラベルとして出す必要はない。あくまで、文章の流れとして自然に伝わるようにする。

### 2. Related Work

Related Work の目的は、先行研究を網羅的に列挙することではない。本研究がどの研究の流れに位置し、既存研究とどの点で異なるのかを明確にすることである。

章全体の流れとして、今回扱う研究目的が明確になり、本研究の位置づけが自然に伝わるようにする。Related Work の最後では、必要に応じて本研究の立ち位置を簡潔に再確認する。ただし、「本研究は既存研究と異なる」とだけ述べるのではなく、何を同じ文脈から引き継ぎ、何を異なる形で扱うのかを明確にする。

### 3. Problem Setting / Problem Statement

Problem Setting では、論文が扱う問題を論理的に定義する。前提、入力、出力、目的、制約、評価対象、実験条件を混同しない。

研究によっては明示的な「想定環境」の節が不要な場合もある。その場合でも、前提、入力、出力、実験条件が文章の中で明確に分かれるようにする。

### 4. Proposed Method / Method

Proposed Method では、提案手法の中核となる考え方を明確に示す。細かい処理や数式に入る前に、Introduction や Problem Setting で示した問題に対して、提案手法がどのような方針で解くのかを説明する。

### 5. Experimental Setup

Experimental Setup では、提案手法がどのような条件で評価されたのかを再現可能な程度に説明する。データセット、比較手法、評価指標、実装条件、学習条件を明確にする。

ただし、すべての研究で上記を網羅する必要はない。データセットを用いない研究、学習を行わない研究、ロボット実機実験を中心とする研究などでは、研究内容に合わせて必要な項目を選ぶ。

### 6. Results / Experimental Results

Results では、数値結果を示すだけでなく、その結果から何がわかったのかを書く。「良い結果が得られた」「性能が向上した」だけでは不十分である。

### 7. Discussion

Discussion では、実験結果を踏まえて、論文の主張を補強する。Results で示した数値を繰り返すだけではなく、その結果が研究全体に対して何を意味するのかを述べる。

限界は率直に書く。ただし、限界を書くだけで終わらせず、その原因と改善方針を述べる。提案手法の有効性を過度に一般化しない。実験で支えられている範囲と、今後検証が必要な範囲を区別する。

### 8. Conclusion

Conclusion では、研究の目的、提案内容、主要な結果、得られた知見を簡潔にまとめる。本文の単なる繰り返しにしない。

貢献を書く場合は、原則として過去形で書く。「作ったもの」だけでなく「示したこと」を含める。実験で支えられていない強い主張は避ける。展望を書く場合も、本文で示した結果と自然につながる範囲に留める。

## 章間接続マップ

```
Introduction → Related Work
  研究目的と問題を示す → その問題が先行研究の流れの中でどこに位置するかを示す

Related Work → Problem Setting
  既存手法の共通制約を示す → その制約を形式的に定義する

Problem Setting → Method
  入力・出力・成功条件を定義する → その定義に対して提案手法がどう解くかを説明する

Method → Experimental Setup
  提案手法を説明する → その手法を何で、どのように評価したかを説明する

Experimental Setup → Results
  評価設定を説明する → その設定での結果を示す

Results → Discussion
  数値・定性結果を示す → それが研究全体にとって何を意味するかを解釈する

Discussion → Conclusion
  結果の意味・限界を論じる → 研究全体の貢献と展望を簡潔にまとめる
```

## エージェントの出力方針

執筆案を出す場合は、単に本文を生成するだけでなく、必要に応じて以下を示す：

- その章や段落の狙い。
- 文章の流れ。
- 本文案。
- 改善意図。
- 残っている懸念。
- 研究内容によって確認すべき追加項目。

**本文案では、内部的な整理語をそのまま出さない。**「背景として」「技術的問題点として」「キーアイデアとして」といった書き方は避ける。自然な論文本文として読める文章を出力する。

レビュー結果を出す場合は、批判だけで終わらせない。どのように直せばよいかを具体的に示す。

チェックリストを使う場合も、単に「未回答」と判定するのではなく、論文の主張を支えるうえで本当に必要な情報かどうかを判断する。研究内容によって不要な項目は、無理に追加させない。
