# CS・ロボット論文 — 執筆ルール

Used by `draft_writer_agent`, `peer_reviewer_agent`.

対応する節の流れ → `cs_paper_section_flow.md`
対応するチェックリスト → `cs_paper_checklist.md`

各ルールは、節の流れ（何を書くか）とは独立して適用する。

---

## 日本語表現の基本ルール

日本語で執筆・レビューする場合は、自然で読みやすい文章を優先する。

| ルール | 説明 |
|--------|------|
| 独自用語を安易に使わない | 一般的に使われない独自用語を避ける。独自の概念名を使う場合は、初出で自然に定義する |
| 係受けを短くする | 係受けが遠い文章にしない |
| 一文一主張 | 一文を短くし、一文に複数の主張を詰め込まない |
| 「必要・重要」止め禁止 | 「〜が重要である」「〜が必要である」だけで終わらせない |
| 内部整理語を本文に出さない | 「技術的問題点として〜」「キーアイデアとして〜」「背景として〜」のような設計メモの見出しを本文に出さない |
| 箇条書き・表への過依存を避ける | 箇条書きや表に頼りすぎず、本文の流れの中で自然に説明する |
| 文型の機械的繰り返しを避ける | 実験結果の説明では、同じ文型を機械的に繰り返さない |
| LLM 造語・抽象語を避ける | LLM が作ったような一般的でない造語や、不自然な抽象語を避ける |

「背景」「目的」「問題」「アイデア」「評価」「限界」などは、執筆時の内部的な整理として使う。実際の論文本文では、これらの語を見出しのように並べるのではなく、読者が自然にその順序で理解できる文章にする。

## Introduction の執筆ルール

### R-INTRO-1: 目的を早く示す

Introduction の早い段階で、本研究が何を目指すのかを述べる。既存研究の説明は、その目的に読者を導くために必要な範囲に留める。

**Bad:** 既存研究にはA、B、Cがあり、それぞれの手法では……。

**Good:** 本研究では、〇〇を実現することを目的とする。この目的に対して、既存の〇〇に基づく方法では、△△を扱うことが難しい。

### R-INTRO-2: 問題は「何ができないのか」として書く

「〜という課題がある」だけでは、問題の実体が伝わりにくい。現在の方法や設定では何ができないのか、どの条件で不十分なのかを具体的に述べる。

**Bad:** 〇〇には技術的問題点がある。

**Good:** 既存手法は〇〇を扱える。一方で、入力に△△が含まれない場合、□□を判断できない。そのため、◇◇の状況では適切な計画が難しくなる。

### R-INTRO-3: 既存手法の不十分さは研究群として抽象化する

既存手法として1つの論文だけを挙げて不十分さを述べると、提案手法が狭い範囲の問題だけを扱っているように見える。また、関連研究の把握が不十分な印象を与える可能性がある。Introduction では、既存手法群が共通して抱える制約や前提を抽象化して述べる。個別研究の詳細は、必要に応じて Related Work に回す。

**Bad:** 手法Aでは〇〇を扱えないため、本研究では新しい手法を提案する。

**Good:** 既存の〇〇に基づく手法は、△△を前提として有効に機能する。一方で、□□のように△△が十分に得られない状況では、◇◇を判断するための情報が不足する。

### R-INTRO-4: 「必要である」ではなく、問題と解決方針の関係を書く

「〜が必要である」という文を多用すると、一般論に見えやすい。何が問題で、それに対して本研究がどのような考え方を導入するのかを書く。

**Bad:** 〇〇を推定する枠組みが必要である。

**Good:** 〇〇だけでは△△を判断できない。本研究では、□□を用いることで、△△を推定する。

### R-INTRO-5: アイデア・実装方法・評価を混同しない

Introduction では、研究の中心的な考え方、具体的な実装方法、実験評価を区別する。

- **中心的な考え方**: 問題をどう捉え、何によって解くのか。
- **実装方法**: その考え方をどのようなモデル、アルゴリズム、システムとして実現するのか。
- **評価**: その方法によって何を検証したのか。

読者が問題と考え方を理解する前に、数式、モジュール名、パラメータ名を出しすぎない。

### R-INTRO-6: 提案手法の差分は主要なものに絞る

Introduction で既存手法との差分を述べる場合は、細かい実装差分を列挙しない。読者が「なぜその差分が必要なのか」を理解できる主要な違いに絞る。差分を述べるだけでなく、その差分を導入することでなぜ問題を解決できると考えられるのかも説明する。

### R-INTRO-7: 貢献は「作ったもの」だけでなく「示したこと」を書く

**Bad:** 新しいシステムを開発した。

**Good:** 〇〇を用いることで、従来扱えなかった△△を計画に反映する方法を提案した。

評価に関する貢献では、性能向上だけでなく、提案手法が有効な条件、限界、今後の改善点も必要に応じて述べる。

### R-INTRO-8: Eye-catch figure を検討する

提案手法全体の構成や研究の狙いを直感的に示す図を置くことが有効である。特に、対象タスク、既存手法との違い、提案手法の全体像が文章だけでは伝わりにくい場合は、図によって読者の理解を助ける。図は単なる装飾ではなく、論文の主張・問題設定・提案手法の中心的な考え方を一目で理解できるものにする。

## Related Work の執筆ルール

### R-RW-1: 先行研究の羅列にしない

Related Work の目的は先行研究を網羅的に列挙することではなく、本研究がどの研究の流れに位置し、既存研究とどの点で異なるのかを明確にすること。

### R-RW-2: 本研究との関係を軸に整理する

各研究群について、何を目的としてきたのか、どのような方法が使われてきたのか、本研究の目的とどのように関係するのかを説明する。

### R-RW-3: 引用数を目的にしない

個別研究の説明は、研究目的との関係がわかる範囲に留める。引用を増やすこと自体を目的にしない。

## Problem Setting の執筆ルール

### R-PS-1: 前提と実験条件を分ける

Problem Setting では、研究が扱う一般的な問題を述べる。実験で用いた具体的な数値、データセット名、試行回数、条件名、パラメータ値は、原則として Experiments に置く。

**Bad:** 本研究では、12種類の条件と1050試行を扱う問題を解く。

**Good:** 本研究では、過去に得られた観測情報を利用できる状況で、現在の入力から計画に必要な情報を推定する問題を扱う。実験で用いた具体的な条件は、Experimental Setup で述べる。

### R-PS-2: 入力と出力を明確にする

手法やタスクを説明するときは、読者が「何を与えると、何が得られるのか」を理解できるようにする。提案手法に固有の入力だけでなく、タスクとして一般に与えられる入力を説明する。

ただし、記号・パラメータ・評価指標・実験条件など、対応関係を正確に示す必要があるものは表や箇条書きを使ってよい。

### R-PS-3: 望ましい解を説明する

出力形式だけでなく、どのような出力が望ましいのかを説明する。タスクの成功条件（例：正しいラベル、安全性・効率性を満たす経路）を示す。

### R-PS-4: 本研究で扱わない範囲を必要な範囲で明確にする

「本研究では〇〇を扱わない」と機械的に列挙するのではなく、論文の主張に必要な範囲で自然に説明する。

### R-PS-5: 初出の用語は自然に定義する

用語集のように箇条書きで並べるのではなく、本文の流れの中で自然に説明する。

**Bad:** Trial: 1回の試行。Condition: 実験条件。

**Good:** 本実験では、ロボットが開始地点から目標地点まで移動する1回の走行を trial と呼ぶ。この単位で到達時間と安全性を評価する。

## Proposed Method の執筆ルール

### R-MTH-1: 冒頭で解決方針を述べる

Method に入ってすぐに詳細なアルゴリズム説明を始めない。まず、提案手法が何を入力とし、どのような処理によって、何を出力するのかを説明する。

### R-MTH-2: システム全体の overview を置く

手法が複数の処理から構成される場合は、冒頭に overview を置く。overview では：入力から出力までの処理の流れ、主な処理段階、各処理段階の役割、どの部分が提案の中心か、既存手法から拡張した部分はどこかを説明する。

### R-MTH-3: 主要モジュールには役割が伝わる名前を付ける

各モジュールについて、動機、役割、入力、出力、構造を説明する。名前だけを先に出して説明を省略してはならない。

### R-MTH-4: 節タイトルは役割がわかる名前にする

**Bad:** Similarity Search / Evidence Accumulation / Costmap Generation

**Good:** Matching Current Observations with Past Records / Estimating the Relevant Context from Matching Results / Generating a Planning Costmap from the Estimated State

専門的な処理名を使う場合でも、節本文の冒頭で、その処理が手法全体の中で果たす役割を説明する。

### R-MTH-5: 入力・予測・損失関数は明確に定義する

入力を x と置く場合は、x が何を表し、どのような次元や構造を持つのかを説明する。数式は、本文の説明を置き換えるものではない。数式の前後で、その数式が何を意味し、手法全体の中で何のために使われるのかを説明する。

### R-MTH-6: 既存部分と提案部分を区別する

本文で「ここは既存部分である」「ここは提案部分である」と機械的に書く必要はない。読者が自然に区別できるように、導入の仕方と説明順序を工夫する。

### R-MTH-7: 一般性を必要に応じて説明する

提案手法が既存手法の拡張である場合、その拡張が特定の手法だけに依存するものなのか、他の手法にも適用可能な考え方なのかを説明する。一般性を主張する場合は、実験や理論的説明によって支えられる範囲に留める。

## Experimental Setup の執筆ルール

### R-EXP-1: 実験条件は本文の流れの中で説明する

条件や用語を箇条書きで並べるだけでは不十分である。各実験が何を検証するためのものかを本文の流れの中で説明する。

### R-EXP-2: 表を適切に使う

ハイパーパラメータ、データセット統計、実験条件、比較手法、評価指標など、対応関係を正確に示す必要があるものは表で整理してよい。ただし、表を置くだけで説明を終えてはならない。

### R-EXP-3: 標準データセットや標準設定との関係を説明する

分野に標準データセットや標準的な評価設定がある場合、それを用いたかどうかを説明する。用いなかった場合は、研究目的に対してなぜ独自設定が必要だったのかを述べる。

## Results の執筆ルール

### R-RES-1: 表の意味を本文で説明する

何が行で、何が列で、数字が何を意味するのかを説明する。平均や標準偏差を示す場合は、何回の試行に基づく値なのかを述べる。

### R-RES-2: 結果は絶対値で示す

「提案手法はベースラインより10%向上した」といった相対的な説明だけでは不十分である。ベースラインと提案手法の絶対的な性能値を示したうえで、その差が何を意味するのかを説明する。

### R-RES-3: 評価尺度の理由を説明する

評価尺度は、論文の主張と対応している必要がある。単に慣習的に使われているからではなく、本研究の目的を評価するうえでなぜ妥当なのかを説明する。複数の評価尺度がある場合、どれが主要な尺度で、どれが補助的な尺度なのかを明確にする。

### R-RES-4: 定性的結果は解釈まで書く

成功例や失敗例を示す場合、結果を提示するだけでは不十分である。Ground Truth、ベースライン、提案手法の出力を比較し、どの点が適切だったのか、またはなぜ失敗したのかを説明する。読者に解釈を丸投げしない。

### R-RES-5: Ablation study は目的を明確にする

何を検証するために何を取り除いたのかを説明する。Ablation のモデルには、必要に応じて Model (i), Model (ii), … のような名前を付ける。

**Bad:** 提案手法(i)では〇〇を除いた。

**Good:** 提案手法から〇〇を除いたモデルを Model (i) とする。この比較により、〇〇が△△に与える影響を調べる。

### R-RES-6: 失敗例を分析する

入力、特徴抽出、推定、中間表現、計画、制御、評価指標など、どの段階に原因があるのかを切り分ける。必要に応じて、失敗例を人手でカテゴリに分類し、各カテゴリの定義を示す。主要なボトルネックと possible solution を説明する。

## 禁止事項

以下は、いかなる場合も避ける：

**構成・論理**
- 目的を示す前に既存研究の詳細を長く説明する。
- 既存手法を1つだけ引用して、その不十分さだけから提案の必要性を説明する。
- アイデア、実装方法、評価方法を混同する。
- Related Work を先行研究の羅列にする。
- Problem Setting に実験固有の条件を入れすぎる。

**Method の書き方**
- Method の冒頭から詳細な数式や処理名に入る。
- 節タイトルをアルゴリズム名だけにする。
- 初出の用語を定義せずに使う。
- 用語定義を不自然な箇条書きにする。

**Results の書き方**
- 実験結果を「良かった」「性能が向上した」で終わらせる。
- 評価指標を定義せずに結果だけ示す。
- 表や図を置くだけで本文中の解釈を省略する。
- 定性的結果を示すだけで、なぜ成功・失敗したのかを説明しない。
- Ablation study で、何を検証するための比較なのかを説明しない。
- 失敗例や限界を隠す。

**Contribution の書き方**
- Contribution を「作ったもの」だけで書く。
- 限界を書くだけで、原因や改善方針を示さない。

**日本語表現**
- 「課題がある」「必要である」だけで問題を説明する。
- 「技術的問題点として〜」のように、設計メモの語を本文に出す。
- 係受けが遠い長文を書く。
- 一文に複数の主張を詰め込む。
- LLM が作ったような一般的でない造語を使う。

**その他**
- 研究内容に合わないチェック項目を無理に本文へ入れる。
